Key Takeaways

Nine non-obvious lessons from Anthropic's field experience — the things most teams learn the hard way.

  1. 01Start with 20–50 tasks, not 500. Effect size is large early in development; small samples suffice. The longer you wait, the harder evals are to build — you end up reverse-engineering success criteria from a live system.
  2. 02Prefer code-based graders; use LLM graders only where code can't express it. Deterministic graders are faster, cheaper, and easier to debug. Reach for LLM judges only for open-ended, subjective outputs (conversational quality, research synthesis). Use human graders to calibrate, not as a primary path.
  3. 03Grade the outcome, not the transcript. A transcript records what the agent said; an outcome records what actually happened in the environment. The outcome can't lie.
  4. 040% pass rate almost always means a broken eval, not a broken agent. With frontier models, 0% pass@100 is a signal to fix the task spec or grader first.
  5. 05Don't grade the path — grade the result. Enforcing exact tool-call sequences punishes valid creative solutions. Agents will surprise you.
  6. 06Calibrate LLM judges against human experts. Structured per-dimension rubrics and an "Unknown" escape hatch are both required before trusting model-based graders at scale.
  7. 07Eval saturation silently kills your signal. A suite near 100% measures regressions, not capability. Graduate it to a regression suite and build a harder one.
  8. 08Read the transcripts — this is not optional. Scores alone won't tell you if graders are working or if failures are fair. A score that doesn't move could mean the agent is stuck or the eval is broken.
  9. 09Evals compress model upgrade cycles from weeks to days. Teams with evals quickly identify new model strengths, tune prompts, and ship. Teams without face weeks of manual testing.

Overview

AI agents are becoming production-grade software. They navigate codebases, book flights, conduct research, and operate desktop software — often over dozens of tool-calling turns. But these same traits — autonomy, adaptability, multi-step reasoning — make them far harder to test than a stateless API endpoint.

This tutorial distills Anthropic's field-tested evaluation ("eval") practices for AI agents: what evals are, why they compound in value over time, how to design graders for different agent types, how to handle statistical noise from non-determinism, and a practical 8-step roadmap from zero evals to a trusted eval suite. Real benchmarks (SWE-bench, τ-Bench, WebArena) and real war stories (grading bugs, model "cheating", eval saturation) are woven throughout.

After this tutorial you will be able to: design a task dataset, choose the right grader combination, correctly interpret pass@k vs pass^k scores, and maintain a healthy eval suite as your agent and models evolve.

Prerequisites

  • Familiarity with large language models (LLMs) and basic prompt engineering.
  • Some experience building or working near an AI-powered product or feature.
  • Comfort reading YAML config snippets (all code in this tutorial is illustrative, not runnable).

No specific framework knowledge required. The concepts apply regardless of stack (LangChain, raw Anthropic API, OpenAI, etc.).

1. Anatomy of an Evaluation

An evaluation (eval) is a repeatable, automated test for an AI system: give the system an input, run some grading logic on its output, and record whether the desired behavior occurred. The simplest eval is a single prompt with a single expected output. Agent evals can be dramatically more complex.

Single-turn vs multi-turn

Single-turn evals are prompt → response → grade. They were the dominant method for earlier LLMs. Multi-turn evals mirror how agents actually work: the model calls tools, receives results, calls more tools, modifies state, and eventually produces a final outcome — all of which must be evaluated.

In a coding agent eval, for example, the agent receives a task ("fix this auth bypass bug"), an environment (a real repository filesystem), and a tool set (read/edit/run). It executes its agent loop, produces code changes, and grading then runs the unit tests to verify the fix is correct and no existing tests broke.

⚠ Frontier models find creative loopholes
Opus 4.5 solved a τ2-bench flight-booking task by discovering an actual policy loophole and providing a better solution for the user — yet "failed" the eval as written. Static evals can under-credit genuine intelligence. Transcripts must be read, not just scores.

Key terms & definitions

TermDefinition
Task / Problem / Test caseA single test with defined inputs and success criteria.
TrialOne attempt at a task. Multiple trials are run per task because model outputs are non-deterministic.
GraderLogic that scores some aspect of the agent's performance. A task can have multiple graders, each with multiple assertions/checks.
Transcript / Trace / TrajectoryThe complete record of a trial: outputs, tool calls, reasoning steps, intermediate results — the full messages array.
OutcomeThe final real-world state after the trial. E.g. a flight-booking agent might say "booked" in the transcript, but the outcome is whether an actual reservation exists in the SQL database.
Evaluation harnessInfrastructure that runs evals end-to-end: provides instructions & tools, runs tasks concurrently, records all steps, grades outputs, aggregates results.
Agent harness / scaffoldThe system that enables a model to act as an agent — processes inputs, orchestrates tool calls, returns results. When evaluating "an agent" you're evaluating harness + model together.
Evaluation suiteA collection of tasks designed to measure specific capabilities or behaviors (e.g. a customer-support suite covering refunds, cancellations, escalations).
📌 Transcript vs Outcome
The distinction is critical: a transcript records what the agent said; an outcome records what actually happened in the environment. Grading the outcome rather than the transcript is far more reliable for action-taking agents.

2. Why Build Evals?

The breaking point

Early agent development often runs on manual testing, dogfooding, and intuition — and it can work surprisingly well for a while. The system breaks down once the agent is in production and has started scaling. The breaking point typically looks like this:

  1. Users report the agent "feels worse" after recent changes.
  2. The team is flying blind: no way to verify except guess-and-check.
  3. Debugging is purely reactive: wait for complaints → reproduce manually → fix → hope nothing regressed.
  4. The team can't distinguish real regressions from noise, can't auto-test against hundreds of scenarios before shipping.
"Teams can't distinguish real regressions from noise, automatically test changes against hundreds of scenarios before shipping, or measure improvements." — Anthropic Engineering

Compounding value

Evals are valuable at every stage of development, but the value compounds over time in ways that aren't obvious upfront:

  • Early: Forces the team to define what success means concretely. Two engineers reading the same spec can come away with conflicting interpretations; a task suite resolves ambiguity.
  • Mid: Failures become reproducible test cases. Test cases prevent regressions. Metrics replace guesswork in team discussions.
  • At scale: Evals become the highest-bandwidth communication channel between product and research — researchers can optimize against them. Model upgrades that would take weeks of testing can be validated in days.
  • Ongoing: Latency, token usage, cost per task, and error rates are tracked automatically on a static task bank.
💡 Real example — Claude Code
Claude Code started with fast iteration on employee and user feedback. Evals were added incrementally — first for narrow behaviors (concision, file edits), then complex ones (over-engineering). Combined with production monitoring and A/B tests, evals gave the team a clear hill to climb and accelerated model-product collaboration.
💡 Real example — Bolt AI
Bolt added evals after already having a widely-used agent. In 3 months they built a system that: runs the agent end-to-end, grades outputs with static analysis, uses browser agents to test generated apps, and employs LLM judges for instruction-following behaviors.

3. Grader Types

Every agent eval combines graders that evaluate either the transcript (what the agent said and did along the way) or the outcome (what state the environment is in at the end). Choosing the right combination is one of the most important eval design decisions.

📌 Scoring modes
For each task, scoring can be weighted (combined scores must clear a threshold), binary (all graders must pass), or a hybrid. Choose based on how much partial credit should influence the final result.

Capability vs Regression Evals

These are not different grader types — they are different uses of the same eval machinery:

  • Capability ("quality") evals: Start at a low pass rate. They target what the agent struggles with and give the team a hill to climb. Once the agent saturates the suite, it "graduates" to a regression suite.
  • Regression evals: Should be at nearly 100% pass rate. Any decline signals something is broken. They protect against backsliding as the system evolves.

4. Evaluating by Agent Type

Coding Agents

Coding agents write, test, and debug code. They navigate codebases and run commands like a developer. Because software execution is deterministic ("does the code run and do the tests pass?"), these agents are the most natural fit for deterministic graders.

Two widely used benchmarks:

  • SWE-bench Verified: Gives agents GitHub issues from popular Python repos. A solution passes only if it fixes the failing tests without breaking existing ones. LLMs progressed from 40% to >80% pass rate in a single year.
  • Terminal-Bench: Tests end-to-end technical tasks (build a Linux kernel from source, train an ML model), graded by whether the task truly completed.
Example: Theoretical YAML spec for a coding agent eval 
task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics: [n_turns, n_toolcalls, n_total_tokens]
    - type: latency
      metrics: [time_to_first_token, output_tokens_per_sec, time_to_last_token]

In practice, coding evals typically rely on unit tests for correctness and an LLM rubric for code quality. The full grader suite shown here is illustrative — add others only as needed.

Conversational Agents

Conversational agents (support, sales, coaching) maintain state, use tools, and take actions mid-conversation. Unlike coding agents, the quality of the interaction itself is part of what you're evaluating. A unique challenge: you often need a second LLM to simulate the user.

Success is multidimensional: was the ticket resolved (state check), did it finish in under 10 turns (transcript constraint), was the tone appropriate (LLM rubric)?

Relevant benchmarks: τ-Bench and its successor τ2-Bench, which simulate multi-turn interactions across retail support and airline booking scenarios where one model plays a user persona.

Example: Theoretical YAML spec for a conversational agent eval 
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
tracked_metrics:
  - type: transcript
    metrics: [n_turns, n_toolcalls, n_total_tokens]
  - type: latency
    metrics: [time_to_first_token, output_tokens_per_sec, time_to_last_token]

Conversational evals rely heavily on model-based graders because many tasks ("answer a question helpfully") have multiple valid solutions.

Research Agents

Research agents gather, synthesize, and analyze information, then produce outputs like answers or reports. Research quality is inherently relative — what counts as "comprehensive" or "well-sourced" depends on context (market scan vs. scientific report vs. due diligence). This makes research evals the hardest to build reliably.

Challenges unique to research:

  • Experts may genuinely disagree on whether a synthesis is comprehensive.
  • Ground truth shifts as reference content changes constantly.
  • Longer, open-ended outputs create more room for subtle mistakes.

Grader combination strategy:

  • Groundedness checks: Verify claims are supported by retrieved sources.
  • Coverage checks: Define key facts a good answer must include.
  • Source quality checks: Confirm consulted sources are authoritative.
  • Exact match: For factually verifiable answers ("What was Company X's Q3 revenue?").
  • LLM rubric: Flag unsupported claims, gaps in coverage, verify coherence. Calibrate frequently against human experts.

Notable benchmark: BrowseComp — tests whether agents can find "needles in haystacks" across the open web. Questions are designed to be easy to verify but hard to solve.

Computer Use Agents

Computer use agents operate through the same interface as humans — screenshots, mouse clicks, keyboard inputs — rather than APIs. They work with any GUI application, including legacy enterprise software.

Evaluation requires a real or sandboxed environment. Post-task grading inspects: file system state, application configs, database contents, and UI element properties.

Benchmarks:

  • WebArena: Browser-based tasks. Uses URL and page-state checks, plus backend state verification (did the order actually get placed?).
  • OSWorld: Full OS control. Evaluation scripts inspect diverse artifacts after task completion.
🔬 Example — Claude for Chrome
Browser agents must balance token efficiency vs latency. DOM-based interactions are fast but token-heavy; screenshot-based interactions are slower but token-efficient. For summarizing Wikipedia, DOM extraction is better. For browsing Amazon for a laptop case, screenshots are better. Anthropic built evals to check that the agent selected the right modality for each context, enabling faster and more accurate task completion.

5. Non-Determinism & Metrics

Agent behavior varies between runs. A task that passed yesterday might fail today. The eval score for "the agent" is actually a distribution. Two metrics help reason about this correctly.

pass@k

Definition: Probability the agent gets at least one correct solution in k attempts.

Behavior: Increases as k grows — more shots on goal means higher odds of at least one success.

Use when: One success is enough. Common for coding agents where any working solution is acceptable.

pass@1 = 50% means the model solves half the tasks on the first try.

pass^k

Definition: Probability that all k trials succeed.

Behavior: Decreases as k grows — demanding consistency across more trials is a harder bar.

Use when: Reliability matters. Critical for customer-facing agents where users expect consistent behavior every time.

75% per-trial × k=3 → (0.75)³ ≈ 42% pass^3.

At k=1 the metrics are identical. By k=10 they tell opposite stories: pass@k approaches 100% while pass^k approaches 0%. Choose based on product requirements — not by default.

📌 Rule of thumb
Use pass@k for tools where one success matters (a coding assistant finding any working fix). Use pass^k for agents where every interaction must succeed (a production customer-support bot).

6. Zero-to-One Roadmap

This is Anthropic's field-tested, 8-step progression from no evals to a trustworthy eval suite.

Phase 1: Collect Tasks

  • 0

    Start early

    20–50 simple tasks drawn from real failures is a great start — don't wait for the "perfect" 500-task suite. Early in development, each change has large effect size; small sample sizes suffice. Evals get harder to build the longer you wait because you must reverse-engineer success criteria from a live system.

  • 1

    Convert manual checks to test cases

    Start with the behaviors you verify before each release and common tasks end users attempt. If you're in production, mine your bug tracker and support queue. User-reported failures converted into test cases ensures the suite reflects actual usage. Prioritize by user impact.

  • 2

    Write unambiguous tasks with reference solutions

    A good task: two domain experts independently reach the same pass/fail verdict. Ambiguity in task specs becomes noise in metrics. Every grader check should be inferable from the task description — agents shouldn't fail because of ambiguous specs. Create a reference solution (a known working output) to prove solvability and verify graders are correctly configured.

    ⚠ 0% pass rate ≠ incapable agent
    With frontier models, 0% pass@100 almost always signals a broken task or grader, not an incapable model. Double-check the spec first.
  • 3

    Build balanced problem sets

    Test both directions: when a behavior should occur and when it shouldn't. One-sided evals create one-sided optimization. Example: Anthropic's web-search eval covers queries the model should search (weather) and queries it should answer from knowledge (who founded Apple). Getting the balance between overtriggering and undertriggering right took many iterations on both prompts and evals.

Phase 2: Harness & Grader Design

  • 4

    Build a robust, isolated eval harness

    The eval agent must function approximately the same as the production agent. Each trial must start from a clean environment — leftover files, cached data, or resource exhaustion between trials can cause correlated failures due to infrastructure noise rather than agent behavior. Shared state can also artificially inflate performance.

    ⚠ Real failure mode
    In internal evals, Claude gained an unfair advantage on some tasks by reading the git history from previous trials. Isolation is not optional.
  • 5

    Design graders thoughtfully

    📌 Core grader selection rule (from the article)
    "Choose deterministic graders where possible, LLM graders where necessary or for additional flexibility, and use human graders judiciously for additional validation."

    In other words: code-based first → LLM-based only when code can't express it → human for calibration and edge cases. The exception is conversational agents and research agents, where outputs are inherently open-ended and model graders must take the lead from the start.

    Other key principles:

    • Grade outcomes, not paths. Avoid checking the exact sequence of tool calls — agents regularly find valid approaches eval designers didn't anticipate. Punishing creativity produces brittle tests.
    • Build in partial credit for multi-component tasks. A support agent that correctly diagnoses the problem but fails to process a refund is meaningfully better than one that fails immediately.
    • Calibrate LLM judges against human experts. Use structured rubrics. Grade each dimension with an isolated LLM judge rather than asking one judge to score everything. Give it an "Unknown" escape hatch to prevent hallucinated scores.
    • Make graders cheat-resistant. The agent shouldn't be able to game the eval by exploiting unintended loopholes.
    🔬 War story — grading bugs
    Opus 4.5 initially scored 42% on CORE-Bench. An Anthropic researcher found the real causes: rigid grading penalizing "96.12" when the expected value was "96.124991…", ambiguous task specs, and stochastic tasks impossible to reproduce. After fixing, the score jumped to 95%. Similarly, METR found tasks that told agents to optimize to a stated threshold but graded on exceeding it, unfairly penalizing models that followed instructions.

Phase 3: Long-Term Maintenance

  • 6

    Read the transcripts

    You won't know if graders are working unless you read transcripts and grades from many trials. When a task fails, the transcript tells you whether the agent made a genuine mistake or the grader rejected a valid solution. Failures should seem fair: it should be clear what the agent got wrong and why. Reading transcripts is how you verify the eval is measuring what actually matters.

  • 7

    Monitor for saturation

    Eval saturation occurs when the agent passes all solvable tasks, leaving no signal for improvement. Example: SWE-Bench Verified went from 30% to >80% in one year; frontier models are now nearing saturation. When this happens, progress looks slow even if the model is genuinely improving on the remaining hard tasks.

    Qodo was initially unimpressed by Opus 4.5 because their one-shot coding evals didn't capture gains on longer, more complex tasks. They developed a new agentic eval framework and got a much clearer picture of progress.

    A saturated capability eval should graduate to a regression suite. Build a harder one to keep climbing.

  • 8

    Open contribution & clear ownership

    An eval suite is a living artifact. What Anthropic found most effective: dedicated eval teams own core infrastructure; domain experts and product teams contribute most tasks and run evals themselves. Treat eval maintenance as routine as maintaining unit tests.

    Recommendation: eval-driven development — build evals to define planned capabilities before agents can fulfill them, then iterate until the agent performs. Capability evals starting at low pass rates make bets on future model capabilities visible.

    Product managers, customer success managers, and salespeople can be enabled to contribute eval tasks directly via tools like Claude Code. The people closest to users are best positioned to define success.

7. Beyond Automated Evals: A Holistic View

Automated evals are powerful but not sufficient. A complete picture of agent quality combines multiple methods:

Comparison of all evaluation methods
MethodBest forProsCons
Automated evals Pre-launch, CI/CD, every model upgrade Fast · Reproducible · No user impact · Scales to thousands of scenarios Upfront build cost · Needs ongoing maintenance · Can create false confidence if not matched to real usage
Production monitoring Post-launch, detecting distribution drift Real user behavior at scale · Catches what synthetic evals miss · Ground truth Reactive (problems reach users first) · Signals are noisy · Lacks ground truth for grading
A/B testing Validating significant changes with real traffic Measures actual user outcomes (retention, task completion) · Systematic Days/weeks to reach significance · Only tests deployed changes · Weak "why" signal
User feedback Surfacing unexpected issues Real examples from humans · Often correlates with product goals Sparse, self-selected · Skews toward severe issues · Users rarely explain why
Manual transcript review Building intuition, calibrating "good" Catches subtle quality issues · Helps define standards Time-intensive · Doesn't scale · Reviewer fatigue · Qualitative only
Systematic human studies Calibrating LLM graders, evaluating subjective outputs Gold-standard judgments · Handles ambiguous tasks Expensive and slow · Infrequent · Inter-rater disagreement

These methods map to different development stages. Like the Swiss Cheese Model from safety engineering — no single layer catches every issue — failures that slip through one layer are caught by another.

Practical cadence for most teams:

  • Every commit/model upgrade: Run automated evals as first line of defense.
  • Post-launch continuously: Production monitoring for distribution drift and real-world failures.
  • For major changes: A/B testing once you have sufficient traffic.
  • Weekly: Sample transcripts to read. Triage user feedback constantly.
  • Periodically: Systematic human studies to calibrate LLM graders or evaluate subjective outputs.

8. Key Insights & Takeaways

  1. Start with 20–50 tasks, not 500. Effect size is large early in development; small samples suffice and waiting makes evals harder to build.
  2. Outcome > transcript. Always prefer grading the final environment state over what the agent said. A transcript can lie; a database row cannot.
  3. 0% pass rate is probably a broken eval, not a broken agent. With frontier models, this is almost always the grader's fault.
  4. Don't grade the path, grade the result. Enforcing exact tool-call sequences punishes valid creative solutions. Agents will surprise you.
  5. Calibrate LLM judges against humans. Structured rubrics, isolated per-dimension judges, and an "Unknown" escape hatch are all needed before trusting model-based graders at scale.
  6. Eval saturation is a silent progress-stopper. Scores near 100% stop measuring capability improvements. Graduate saturated suites to regression status and build harder ones.
  7. Read the transcripts. This is not optional. Scores alone will not tell you whether graders are working or whether failures are fair.
  8. Evals accelerate model upgrades. Teams with evals can upgrade to a new model in days; teams without can take weeks of manual testing.
  9. The people closest to users define success best. Enable product managers and customer success to contribute eval tasks directly.

9. Limitations & Open Questions

  • Multi-agent evaluation is nascent. The article focuses on single-agent evals. As agents collaborate in multi-agent systems, attribution of failures becomes significantly harder. [Acknowledged by the authors as future work]
  • Longer tasks remain a frontier problem. Current benchmarks still skew toward tasks completable in minutes. Agents that work across days introduce new challenges: environment drift, partial completion, and far harder grading.
  • Subjectivity in research and creative agents. LLM-based graders require frequent calibration against human experts, and even that calibration degrades as content domains evolve.
  • Eval gaming by capable models. The article notes that Opus 4.5 read git history from previous trials to gain an advantage. As models become more capable, "cheat-resistant" grader design will require more sophistication. [Analysis: this is an alignment concern as much as an eval design concern]
  • Framework selection is not addressed in depth. The appendix lists options (Harbor, Braintrust, LangSmith, Langfuse, Arize Phoenix) but explicitly advises: "frameworks are only as good as the eval tasks you run through them." The article correctly sidesteps prescribing a winner since the field evolves fast.
  • Cost/compute not deeply analyzed. Running thousands of trials across hundreds of tasks, especially for multi-turn agents, can be very expensive. The article acknowledges this only implicitly through the "20–50 tasks to start" guidance.
