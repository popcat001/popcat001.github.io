Key Takeaways

Seven core lessons from Anthropic's experience building and advising production agent systems.

  1. 01Start simpler than you think you need. Optimize a single LLM call with good retrieval and in-context examples before reaching for multi-agent systems. Add complexity only when it demonstrably improves outcomes.
  2. 02Workflows and agents are distinct architectural choices. Workflows use predefined code paths; agents let the LLM direct its own execution. Choose based on whether the required steps can be predicted in advance.
  3. 03Five composable patterns cover most production needs. Prompt chaining, routing, parallelization, orchestrator-workers, and evaluator-optimizer can be mixed and matched to handle almost any use case.
  4. 04Tools deserve as much engineering as prompts. Your agent-computer interface (ACI) is as important as your human-computer interface (HCI). Poorly specified tools are a leading cause of agent failure.
  5. 05Agents amplify both capability and cost. Their autonomy makes them powerful for open-ended tasks in trusted environments, but errors compound across turns. Always test in sandboxed environments with appropriate guardrails.
  6. 06Prefer direct API calls over frameworks in production. Frameworks speed up prototyping but obscure prompts and responses, making debugging harder. Reduce abstraction as you move toward production.
  7. 07Three principles for reliable agents: simplicity, transparency, thorough tool documentation. Agents that show their planning steps are easier to trust, debug, and improve.

Overview

This tutorial distills Anthropic's practical guidance for building AI agents — drawn from working with dozens of teams across industries deploying LLMs in production. The central thesis: the most successful agent implementations use simple, composable patterns, not complex frameworks.

The article covers when to use agents at all, five workflow patterns that handle most production needs, a framework for thinking about full autonomous agents, and often-overlooked advice on tool design. It also includes two real-world appendices on customer support and coding agents.

After this tutorial you will understand: the difference between workflows and agents, when each pattern is the right tool, how to design robust tool interfaces, and the three principles that make agents reliable in production.

Prerequisites

  • Familiarity with LLM API calls (sending prompts, receiving completions).
  • Basic understanding of tool use / function calling in LLMs.
  • Some experience building or prototyping an AI feature.

No specific framework or language required. All patterns are framework-agnostic; code snippets are illustrative Python.

1. Background & Motivation

LLMs have matured rapidly from single-turn text generators into systems capable of multi-step reasoning, tool use, and environmental interaction. But with that power comes an industry-wide temptation to over-engineer: adding orchestration frameworks, specialized libraries, and complex routing logic before they're needed.

Anthropic observed a consistent pattern across its customer deployments: teams that started simple and added complexity deliberately outperformed teams that started with elaborate frameworks. The failures almost always came from complexity that obscured what the model was actually doing.

The article exists to name the composable patterns that actually work, give developers a vocabulary for discussing them, and establish a decision framework for when to use each one.

⚠ The complexity trap

Frameworks abstract away prompts and responses. That makes them easy to start with — and hard to debug when something goes wrong. Incorrect assumptions about what's "under the hood" are a leading source of production failures.

2. Core Concepts

Agentic Systems

The term "agent" means different things to different teams. Anthropic's umbrella term is agentic systems: any system where an LLM takes actions beyond producing a single response — calling tools, reading from memory, modifying external state, or chaining multiple LLM calls.

Workflows vs Agents

Within agentic systems, the most important architectural distinction is:

TypeWho controls execution?Use when
Workflow Predefined code paths — the developer decides the steps and their order at design time. The task can be cleanly decomposed; you want predictability and consistency.
Agent The LLM dynamically decides what to do next based on its current context. The required steps can't be predicted in advance; you need flexibility and model-driven decisions at scale.
📌 Start with neither

For many tasks, a well-optimized single LLM call with retrieval and in-context examples is sufficient. Build upward from there only when you have evidence that it's not enough.

The Augmented LLM — The Core Building Block

Every pattern in this article builds on a single foundational unit: an LLM enhanced with three augmentations.

graph LR U([User / System]) --> LLM[LLM] LLM --> T[Tools\ne.g. search, code exec, APIs] LLM --> M[Memory\nshort-term context + long-term storage] LLM --> R[Retrieval\nRAG / vector search] T --> LLM M --> LLM R --> LLM
AugmentationWhat it providesExample
ToolsAbility to take actions in external systemsWeb search, code execution, database writes
MemoryPersistence across turns (short-term) and sessions (long-term)Conversation history, user profile store
RetrievalAccess to external knowledge at inference timeRAG over a knowledge base, file system reads

The Model Context Protocol (MCP) is Anthropic's open standard for wiring up tools and retrieval sources with a single client implementation, enabling a growing ecosystem of third-party integrations.

3. The Five Workflow Patterns (+ Full Agents)

These patterns are composable building blocks. Real systems mix and nest them. The key is choosing the simplest pattern that solves your problem — then stopping there.

① Prompt Chaining

Sequential steps, each building on the last

② Routing

Classify input → direct to specialised handler

③ Parallelization

Divide work or run multiple attempts simultaneously

④ Orchestrator-Workers

Central LLM dynamically delegates to worker LLMs

⑤ Evaluator-Optimizer

Generate → evaluate → refine in a loop

⑥ Full Agents

LLM directs its own tool use until task is complete

① Prompt Chaining

What it is: Decompose a task into a fixed sequence of LLM calls, feeding each output into the next. Programmatic "gate" checks can verify intermediate results and abort early if something is off-track.

graph LR I([Input]) --> A[LLM Step 1] A --> G{Gate OK?} G -- Yes --> B[LLM Step 2] G -- No --> E([Error / Retry]) B --> C[LLM Step 3] C --> O([Output])

Trade-off: Higher accuracy (each step is simpler) at the cost of higher latency (sequential execution).

✅ Use when
  • The task cleanly decomposes into ordered subtasks.
  • Intermediate outputs are verifiable (programmatic gate checks make sense).
  • You're willing to trade latency for per-step accuracy.
Real examples
  • Generate marketing copy → gate: does it mention all required features? → translate into target language.
  • Write document outline → gate: does outline meet length and structure criteria? → write full document from outline.

② Routing

What it is: Classify the input (via LLM or a traditional classifier) and direct it to a specialized downstream handler — a different prompt, model, or toolset optimized for that input type.

graph TD I([Input]) --> C[Classifier LLM / model] C --> R1[Handler A\nRefund prompt] C --> R2[Handler B\nTech support prompt] C --> R3[Handler C\nSmall model — simple Q&A]

Why it matters: Optimizing a prompt for one type of input often degrades performance on others. Routing lets you maintain separate, specialized prompts without compromising overall quality.

✅ Use when
  • Input types are clearly distinct and benefit from different handling.
  • Classification is reliable (high-confidence labels).
  • You want cost-efficiency: route easy/common queries to cheaper models, hard/rare ones to more capable models.
Real examples
  • Customer service: route refund requests, technical issues, and general questions to separate prompts and toolsets.
  • Cost tiering: route common questions to Claude Haiku 4.5; route rare, complex questions to Claude Sonnet 4.5.

③ Parallelization

What it is: Run multiple LLM calls simultaneously and aggregate results. Two distinct sub-patterns:

Sub-patternWhat happensWhen to use
Sectioning Break the task into independent subtasks; run each in a separate LLM call in parallel. Subtasks are independent. Speed matters. Each aspect benefits from focused attention.
Voting Run the same task multiple times with different prompts/seeds; aggregate or vote on results. You need high-confidence outputs. False positive/negative tradeoffs matter.
graph LR subgraph Sectioning I1([Task]) --> P1[LLM: Part A] I1 --> P2[LLM: Part B] I1 --> P3[LLM: Part C] P1 & P2 & P3 --> AG1[Aggregator] end subgraph Voting I2([Task]) --> V1[LLM: Attempt 1] I2 --> V2[LLM: Attempt 2] I2 --> V3[LLM: Attempt 3] V1 & V2 & V3 --> VT[Vote / Consensus] end
✅ Use when
  • Sectioning: Multiple independent concerns need attention (e.g. content screening + response generation run as separate instances to avoid interference).
  • Voting: High-stakes decisions where you want multiple "opinions" — e.g. code vulnerability review, content moderation with tunable thresholds.

④ Orchestrator-Workers

What it is: A central "orchestrator" LLM receives the task, dynamically decomposes it into subtasks at runtime, dispatches them to "worker" LLMs, and synthesizes the results. The key difference from parallelization: subtasks are not predefined — the orchestrator decides them based on the specific input.

graph TD T([Task]) --> O[Orchestrator LLM] O -- "Subtask A" --> W1[Worker LLM 1] O -- "Subtask B" --> W2[Worker LLM 2] O -- "Subtask C" --> W3[Worker LLM 3] W1 & W2 & W3 --> O O --> R([Result])
✅ Use when
  • You can't predict how many subtasks or what kind will be needed at design time.
  • The scope of work varies significantly by input (e.g. a codebase change that might touch 2 files or 20).
Real examples
  • Coding products making complex changes across multiple files — the orchestrator decides which files need changes and what kind.
  • Multi-source research tasks where the number and nature of sources to consult depends on the topic.

⑤ Evaluator-Optimizer

What it is: One LLM generates a response; a second LLM evaluates it against criteria and provides feedback; the first iterates. The loop continues until quality criteria are met or a maximum number of rounds is reached.

graph LR T([Task]) --> G[Generator LLM] G --> E[Evaluator LLM] E -- "Feedback: needs improvement" --> G E -- "Criteria met" --> O([Output])

The two signs of good fit (directly from the article):

  1. LLM responses can be demonstrably improved when a human articulates feedback — meaning the feedback signal is real and actionable.
  2. The LLM can provide such feedback reliably — meaning the evaluator role is within the model's capability.
✅ Use when
  • Clear evaluation criteria exist and can be expressed to an LLM.
  • Iterative refinement produces measurable value (like polishing a translation or a persuasive essay).
  • You want automatic quality control without human review on every output.
Real examples
  • Literary translation: translator LLM produces a draft; evaluator LLM flags nuance misses and cultural gaps; translator revises.
  • Complex research tasks: generator searches and synthesizes; evaluator decides whether additional searches are warranted.

⑥ Full Agents

What it is: The LLM runs in an open-ended loop, deciding what tool to call next based on the current state of the environment. It operates until the task is complete, a stopping condition is reached, or it requests human input.

graph TD H([Human / System]) -- "Task + tools" --> A[Agent LLM] A --> TC[Tool Call] TC --> E[Environment\ncode exec / APIs / filesystem] E -- "Result / new state" --> A A -- "Need clarification?" --> H A -- "Task complete / max turns" --> O([Output])

Agents are typically just LLMs using tools in a loop based on environmental feedback. That simplicity is intentional — the complexity lives in tool design and guardrails, not in the agent loop itself.

⚠ Costs and risks scale with autonomy
  • Each turn adds latency and token cost.
  • Errors can compound — a wrong early assumption shapes all subsequent steps.
  • Test extensively in sandboxed environments before production.
  • Include stopping conditions (max iterations) to maintain control.
✅ Use when
  • The number or nature of required steps cannot be predicted at design time.
  • The task is open-ended and benefits from adaptive decision-making.
  • You have sufficient trust in the model's judgment and robust guardrails in place.
Anthropic's own agent implementations
  • SWE-bench agent: Resolves real GitHub issues — edits files, runs tests, iterates — with only the PR description as input.
  • Computer use reference implementation: Claude controls a computer via screenshots, clicks, and keyboard to accomplish GUI-based tasks.

4. Prompt Engineering Your Tools (ACI)

No matter which pattern you use, tools are central to agentic systems. Anthropic spent more time optimizing tools than the overall prompt when building their SWE-bench agent. Tool definitions and specifications deserve the same care as your main prompts.

"One rule of thumb is to think about how much effort goes into human-computer interfaces (HCI), and plan to invest just as much effort in creating good agent-computer interfaces (ACI)." — Building Effective Agents

Format selection principles

  • Give the model enough tokens to think before it writes itself into a corner. Avoid formats that require committing to a value early (e.g. diff chunk headers that need a line count before the new code is written).
  • Keep formats close to natural internet text. Markdown code blocks are more natural than JSON-escaped strings for code; the model has seen far more of the former.
  • Eliminate formatting overhead. Don't ask the model to track an accurate line count of thousands of lines, or to escape quotes inside JSON strings it writes.

Tool definition best practices

PracticeWhy it matters
Include example usage in the descriptionModels treat tool docs like they treat docstrings — example input/output is the fastest path to correct use.
Document edge cases and input format requirementsAmbiguous edge cases are where agents fail most often.
Clearly distinguish similar tools from each otherModels get confused when two tools have overlapping scopes and similar names.
Poka-yoke (mistake-proof) the argumentsChange parameter structure so common mistakes become impossible or obvious.
Test with many example inputsUse a workbench to observe what mistakes the model actually makes, then iterate on the definition.
🔬 War story — absolute vs relative filepaths

In the SWE-bench agent, the model made consistent mistakes with tools using relative filepaths after the agent had changed directories. The fix: change the tool to require absolute filepaths. The model then used it flawlessly. One parameter constraint eliminated a whole class of failures.

The model's-eye-view test

Read your tool definition as if you were the model. Ask: "Is it obvious how to use this tool from the description and parameters alone, without having to think hard?" If the answer is no, rewrite the definition. A good tool definition includes:

  • A one-sentence description of what the tool does.
  • Each parameter with its type, purpose, and constraints.
  • At least one example call with expected output.
  • Any edge cases or things to avoid.
  • Clear distinction from any similar tools.

5. Worked Examples

A. Customer Support Agent

Customer support is a natural fit for agents because support interactions require both conversation and action — and success is cleanly measurable (was the issue resolved?).

Architecture walkthrough

Pattern used: Routing (classify query type) + Full Agent (handle the classified query with tools).

  1. Routing layer: Classify the incoming message — refund request, technical support, general question, escalation.
  2. Agent loop for each type:
    • Pull customer data, order history, and knowledge base articles via tools.
    • Decide action: answer from knowledge, issue refund, update ticket, escalate.
    • Execute action programmatically; confirm with customer.
  3. Stopping condition: User-confirmed resolution, or max turns → human handoff.

Why agents add value here: Conversation flow is inherently variable (the customer might ask follow-ups, dispute a decision, provide new information), so hardcoded workflows break quickly. The agent's ability to adapt its next step based on the customer's reply is essential.

📌 Business model signal

Several companies built on this pattern charge only for successful resolutions. That pricing model is only viable if the agent reliably works — a strong real-world endorsement of the pattern's effectiveness.

B. Coding Agent (SWE-bench)

Software development is one of the highest-value domains for agents because code is objectively verifiable: does it run and do the tests pass?

Architecture walkthrough

Pattern used: Full Agent with evaluator-optimizer-style test feedback loop.

  1. Input: A GitHub issue or PR description.
  2. Agent loop:
    • Read relevant files, understand the codebase.
    • Hypothesize a fix; edit the appropriate files.
    • Run the test suite; read the results.
    • If tests fail: diagnose, revise, repeat.
    • If tests pass: verify nothing else broke; submit.
  3. Grounding: Test results are the ground-truth feedback signal at each step — the agent can't fool itself about whether the fix works.

Key insight: Human review remains necessary even when tests pass. Tests verify functional correctness; humans verify that the solution fits the broader system design and intent.

6. Key Insights & Takeaways

  1. 01Complexity is a last resort, not a starting point. Every additional LLM call adds latency, cost, and potential failure modes. Add them only when simpler approaches demonstrably fall short.
  2. 02The augmented LLM is genuinely powerful on its own. Before building multi-agent pipelines, exhaust what a single call with good retrieval, tools, and in-context examples can do.
  3. 03Patterns are not prescriptive. They're a vocabulary and a starting point. Real systems mix, nest, and customize them. Measure performance and iterate.
  4. 04Tool quality is often the binding constraint. Anthropic spent more time on tool definitions than on the agent prompt in their SWE-bench implementation. Treat tool engineering as first-class work.
  5. 05Human oversight is not optional for high-stakes agents. Checkpoints, max-iteration stops, and sandboxed testing are not just best practices — they're the difference between a useful agent and a runaway one.
  6. 06Ground truth from the environment is what makes agents work. At every step, the agent must observe real feedback — test results, API responses, database state — not just its own reasoning.

The article closes with three core principles — worth displaying prominently:

① Simplicity — keep agent design minimal ② Transparency — show planning steps explicitly ③ Tool documentation — thorough ACI engineering

7. Limitations & Open Questions

  • Pattern selection guidance is qualitative. The article gives heuristics for when to use each pattern, but no quantitative thresholds. Teams still need to measure and experiment to find the right choice for their use case. [Analysis]
  • Multi-agent coordination is not addressed. The article focuses on single-agent architectures. As systems involve multiple cooperating agents with shared state, new failure modes emerge (race conditions, conflicting decisions, attribution). These are acknowledged but not covered.
  • Error recovery is underspecified. The article recommends sandboxed testing and guardrails but doesn't describe concrete patterns for error detection, rollback, or graceful degradation inside an agent loop. [Analysis: this is likely intentional given the post's scope]
  • Framework advice may age quickly. The specific frameworks listed (Claude Agent SDK, Strands, Rivet, Vellum) and the advice to prefer direct API calls were accurate as of December 2024. The ecosystem evolves fast.
  • The "simplest solution first" principle requires measurement. Without evals, teams can't know when a simpler solution is "good enough." This post pairs naturally with Anthropic's evals guide, which provides the measurement infrastructure this principle assumes. [Analysis]

8. Glossary

Expand glossary (A–Z)
TermDefinition
ACI (Agent-Computer Interface)The set of tool definitions, parameter schemas, and documentation that mediates between an agent and the systems it controls. Analogous to HCI for humans.
Agentic systemAny system where an LLM takes actions beyond producing a single response — calling tools, chaining calls, modifying state, or maintaining memory across turns.
Agent (strict definition)An agentic system where the LLM dynamically directs its own process and tool usage, rather than following a predefined code path.
Augmented LLMAn LLM enhanced with tools, memory, and retrieval — the foundational building block of all agentic patterns.
Evaluator-optimizerA workflow where one LLM generates output and a second evaluates it, looping until quality criteria are met.
GateA programmatic check in a prompt chaining workflow that verifies an intermediate output meets criteria before passing it to the next step.
Ground truthReal environmental feedback (test results, API responses, database state) that the agent uses to assess its own progress — as opposed to self-generated reasoning.
MCP (Model Context Protocol)Anthropic's open standard for connecting LLMs to tools and data sources via a single client implementation.
Orchestrator-workersA workflow where a central LLM dynamically decomposes a task and delegates subtasks to worker LLMs. Subtasks are determined at runtime, not design time.
ParallelizationRunning multiple LLM calls simultaneously. Two sub-patterns: Sectioning (independent subtasks in parallel) and Voting (same task multiple times for consensus).
Poka-yokeMistake-proofing a tool's arguments so that common errors are structurally impossible or immediately obvious.
Prompt chainingA workflow that decomposes a task into sequential LLM calls, each processing the previous one's output, with optional gate checks.
RoutingA workflow that classifies an input and directs it to a specialized handler (different prompt, model, or toolset).
SectioningA parallelization sub-pattern where a task is broken into independent subtasks run simultaneously.
Stopping conditionA rule that terminates an agent loop — e.g. task complete, max iterations reached, or confidence below threshold. Critical for preventing runaway agents.
VotingA parallelization sub-pattern where the same task runs multiple times and results are aggregated or voted on for higher confidence.
Workflow (strict definition)An agentic system where LLMs and tools are orchestrated through predefined code paths decided at design time.
